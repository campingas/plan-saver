## Product

Plan-Saver is a private archive for HTML plans and reports generated by the `html-planning` workflow.

- Magic-link authentication is restricted by `ALLOWED_EMAILS` and delivered through Resend.
- Bearer API tokens ingest plans and reports through `POST /api/v1/documents`; tokens are hashed at rest, can be revoked, and are shown once alongside a filled machine-setup command when created.
- Documents are grouped by project and slug, with separate plan/report kinds and append-only numbered versions.
- Authenticated users can browse projects, inspect versions, download HTML, issue share links, and revoke tokens or shares.
- Owners can delete a single version, a whole document, or a whole project; deletes cascade to child rows, deleting the last version also removes its document, and project deletion requires typing the project slug.
- A share link grants access to one version without requiring an authenticated session.

## Data and security

- Next.js App Router pages and route handlers use Drizzle ORM with Supabase Postgres.
- Runtime connections use the transaction pooler through `DATABASE_URL`; migrations prefer `DIRECT_URL` through the session pooler.
- Application tables retain owner relationships, and application reads and mutations enforce user or share-token authorization.
- Every table has RLS enabled with no policies, blocking Supabase Data API roles while the application database role performs server-side access.
- Stored HTML is returned only by the dedicated viewer route with a restrictive CSP, private no-store caching, and `X-Robots-Tag: noindex`; display responses add local syntax highlighting while raw downloads preserve the archived source.
- Embedded documents run in an iframe sandbox with scripts allowed and same-origin access withheld.

## Repository and operations

- Application code lives under `src/app`, shared server behavior under `src/lib`, and schema/database setup under `src/db`.
- Drizzle migrations and metadata are tracked under `drizzle`.
- The supported local workflow uses Bun, with lint, explicit typecheck, native tests, production build, Drizzle consistency, and diff checks as validation gates.
- Vercel configuration targets the `sin1` region; live deployment health and remote migration state are not recorded in the repository.
- `bun run test` uses a disposable PostgreSQL 17 container by default, applies all migrations, and removes the database container after the native Bun suite.
