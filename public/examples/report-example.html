TL;DR: The magic-link flow was exercised end to end. Sign-in is correctly restricted to allowlisted emails (403 otherwise), links are single-use and short-lived, and the session cookie is HttpOnly + SameSite=Lax. One recommendation: verify a sending domain so login emails stop flirting with the spam folder.

Findings

1. Non-allowlisted addresses cannot request a link

The guard lives in sendMagicLink, before any email is sent, so unknown addresses generate no traffic at all. Observed: 403 This email is not allowed to sign in.

2. Used and expired links fail closed

Re-visiting a consumed link lands on the login page with an error, never a session. Verification rows are cleaned up after use.

Recommendations

  • Verify a sending domain: removes the onboarding-sender spam risk and unlocks any recipient.
  • Open: session lifetime is the Better Auth default; shorten if the archive ever holds sensitive material.
